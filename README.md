# Практическое занятие №11 - GraphQL API с использованием gqlgen

## Описание

Реализация **GraphQL** сервиса на Go с использованием библиотеки **gqlgen**, который предоставляет операции запросов (Query) и изменений (Mutation) для управления задачами (Tasks). Сервис подключается напрямую к базе данных **PostgreSQL** и работает параллельно с **REST API**, что позволяет сравнить оба подхода.

## Контрольные вопросы

### 1. Что такое over-fetching и under-fetching?

**Over-fetching** — получение большего количества данных, чем необходимо, например, когда нужен только `title`, но вместе с ним приходит и `description`.

**Under-fetching** — получение недостаточного количества данных, из-за чего приходится делать дополнительные запросы, например, когда список задач приходит без авторов и для каждой задачи нужен отдельный запрос.

### 2. Почему GraphQL может быть опасен без ограничений?

Без ограничений клиент может отправить чрезмерно вложенный запрос, например `друзья { друзья { друзья { ... } } }`, который потребляет много ресурсов сервера. Это может привести к перегрузке CPU, DoS-атакам и падению сервиса.

### 3. Почему кэширование REST обычно проще?

В REST каждый ресурс обычно имеет уникальный URL, например `/tasks/123`, поэтому можно использовать стандартный HTTP-кэш: браузерный, прокси или CDN.

В GraphQL все запросы обычно идут на один endpoint, а данные определяются телом запроса, поэтому кэширование на уровне HTTP становится сложнее.

### 4. Как отличаются ошибки REST и GraphQL?

| Аспект | REST | GraphQL |
|---|---|---|
| Код HTTP | Семантический: 200, 400, 404, 500 | Обычно всегда 200 |
| Место ошибки | В статусе HTTP | В поле `errors` в JSON |
| Частичный успех | Обычно нет | Возможен: часть данных и часть ошибок |

### 5. В каких случаях вы бы выбрали REST, а в каких GraphQL?

**REST**:
- Публичное API для сторонних разработчиков.
- Простые CRUD-операции.
- Кэширование критически важно.
- Команда не готова к дополнительной сложности.

**GraphQL**:
- Сложные дашборды с множеством представлений.
- Мобильные приложения, где важен трафик.
- Микросервисная архитектура с агрегацией данных.
- Быстрое прототипирование при меняющихся требованиях.
