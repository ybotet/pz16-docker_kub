## 1. Описание работы

### 1.1. Цель

Реализовать **горизонтальное масштабирование** сервиса `tasks` путём запуска нескольких экземпляров (реплик) и распределения трафика между ними с использованием **NGINX** в качестве балансировщика нагрузки (Load Balancer). Необходимо продемонстрировать:

- Способность распределять запросы между несколькими экземплярами.
- Отказоустойчивость при остановке одной или нескольких реплик.
- Важность **stateless** (отсутствия состояния) сервиса для горизонтального масштабирования.

### 1.2. Используемые технологии

| Технология     | Версия   | Назначение                            |
| -------------- | -------- | ------------------------------------- |
| Go             | 1.25     | Язык программирования сервиса `tasks` |
| NGINX          | Alpine   | Балансировщик нагрузки                |
| Docker         | 24+      | Контейнеризация                       |
| Docker Compose | 2.x      | Оркестрация сервисов                  |
| PostgreSQL     | 15       | Общая база данных                     |
| Redis          | 7-alpine | Распределённый кэш                    |

### 1.3. Ключевые понятия

- **Горизонтальное масштабирование:** Увеличение мощности системы путём добавления новых экземпляров сервиса, а не увеличения ресурсов одного экземпляра (вертикальное масштабирование).
- **Балансировщик нагрузки (Load Balancer):** Компонент, распределяющий входящие запросы между несколькими серверами.
- **Stateless:** Сервис не хранит состояние в локальной памяти, что позволяет любому экземпляру обрабатывать любой запрос.
- **Health endpoint:** Точка проверки, указывающая, работает ли экземпляр сервиса.

```bash
# Генерация самоподписанного сертификата
```

## 5. Конфигурация реплик

### 5.1. docker-compose.yml (фрагмент)

```yaml
services:
  tasks_1:
    build:
      context: ../..
      dockerfile: services/tasks/Dockerfile
    container_name: pz7-tasks-1
    environment:
      - AUTH_GRPC_ADDR=auth:50051
      - TASKS_PORT=8082
      - DB_HOST=postgres
      - DB_PORT=5432
      - DB_USER=tasksuser
      - DB_PASSWORD=taskspass
      - DB_NAME=tasksdb
      - REDIS_ADDR=redis:6379
      - INSTANCE_ID=tasks-1
    networks:
      - pz7-network
    depends_on:
      - auth
      - postgres
      - redis
    restart: unless-stopped

  tasks_2: # Аналогичная конфигурация с INSTANCE_ID=tasks-2 # ...

  tasks_3: # Аналогичная конфигурация с INSTANCE_ID=tasks-3 # ...
```

### 5.2. Обоснование

- Отсутствие открытых портов: Реплики не открывают порты на хосте, обмениваясь данными только через внутреннюю сеть Docker.
- Общие ресурсы: Все реплики используют одну базу данных PostgreSQL и один Redis, обеспечивая согласованность данных.
- Уникальные идентификаторы: Переменная INSTANCE_ID позволяет различать, какая реплика обработала запрос.

## 6. Конфигурация NGINX

### 6.1. nginx.conf

```nginx
events {}

http {
log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
'"$http_user_agent" "$http_x_request_id"';

    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log debug;

    # Upstream с несколькими репликами (Round-Robin по умолчанию)
    upstream tasks_service {
        server tasks_1:8082;
        server tasks_2:8082;
        server tasks_3:8082;
    }

    upstream auth_service {
        server auth:8081;
    }

    server {
        listen 8443 ssl;
        server_name localhost;

        ssl_certificate /etc/nginx/tls/cert.pem;
        ssl_certificate_key /etc/nginx/tls/key.pem;
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers HIGH:!aNULL:!MD5;

        # Маршрут для tasks с балансировкой
        location ~ ^/v1/tasks {
            proxy_pass http://tasks_service$request_uri;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto https;
            proxy_set_header X-Request-ID $http_x_request_id;
            proxy_set_header Authorization $http_authorization;
        }

        # Health check для tasks
        location /health/tasks {
            proxy_pass http://tasks_service/health;
        }

        # Маршрут для auth
        location /v1/auth/ {
            proxy_pass http://auth_service;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto https;
            proxy_set_header X-Request-ID $http_x_request_id;
        }

        # Общий health check
        location /health {
            return 200 "OK\n";
            add_header Content-Type text/plain;
        }
    }

}
```

### 6.2. Алгоритм балансировки

Используется алгоритм Round Robin (настройка по умолчанию в NGINX), который последовательно распределяет запросы между доступными репликами.

## 7. Middleware идентификации экземпляра

### 7.1. Реализация в main.go

```go
package main

import (
"net/http"
"os"
// ... другие импорты
)

func main() {
// Чтение идентификатора экземпляра из переменной окружения
instanceID := os.Getenv("INSTANCE_ID")
if instanceID == "" {
instanceID = "unknown"
}

    // Логирование запуска
    log.Printf("[INFO] Запуск экземпляра: %s", instanceID)

    // Middleware, добавляющий заголовок X-Instance-ID
    instanceMiddleware := func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            w.Header().Set("X-Instance-ID", instanceID)
            next.ServeHTTP(w, r)
        })
    }

    // Роутер и middleware
    r := mux.NewRouter()

    // Применение middleware ПЕРЕД остальными
    r.Use(instanceMiddleware)
    r.Use(middleware.RequestID)
    r.Use(middleware.Logging(log))
    r.Use(internalMiddleware.SecurityHeadersMiddleware)

    // ... остальная конфигурация

}
```

### 7.2. Назначение

Заголовок X-Instance-ID позволяет:

- Визуально проверять, какая реплика обработала запрос.
- Отлаживать проблемы балансировки.
- Подтверждать корректное распределение трафика.

## 9. Контрольные вопросы

1. **Что такое горизонтальное масштабирование и чем оно отличается от вертикального?**

| Горизонтальное масштабирование                       | Вертикальное масштабирование                            |
| ---------------------------------------------------- | ------------------------------------------------------- |
| Добавление новых экземпляров сервиса                 | Увеличение ресурсов одного экземпляра (CPU/RAM)         |
| Пример: 3 сервера `tasks` вместо 1                   | Пример: 8GB RAM → 16GB RAM                              |
| Преимущества: отказоустойчивость, отсутствие простоя | Преимущества: проще, без изменений кода                 |
| Недостатки: требуется балансировщик, `stateless`     | Недостатки: физические ограничения, единая точка отказа |

В реализованной работе: применено горизонтальное масштабирование — добавлены 3 реплики `tasks` с NGINX в качестве балансировщика.

2. **Зачем нужен балансировщик нагрузки (load balancer)?**

Балансировщик нагрузки необходим для:

- Распределения трафика: равномерное распределение запросов между доступными экземплярами.
- Отказоустойчивости: автоматическое исключение недоступных экземпляров.
- Прозрачности: клиенты не зависят от конкретных экземпляров.
- Масштабируемости: возможность добавлять/удалять экземпляры без изменения клиентского кода.
- Обслуживания: проведение обновлений без простоя (`rolling updates`).

В реализации: NGINX выступает единой точкой входа, распределяя запросы между `tasks_1`, `tasks_2` и `tasks_3`.

3. **Почему сервис должен быть `stateless` для горизонтального масштабирования?**

Для корректной работы горизонтального масштабирования сервис должен быть `stateless` (не хранить состояние в локальной памяти):

| Хранение состояния в памяти (плохо) | Общее хранилище (хорошо)     |
| ----------------------------------- | ---------------------------- |
| Сессии локально                     | Сессии в Redis/БД            |
| Локальный кэш                       | Централизованный кэш (Redis) |
| Временные файлы локально            | Общее хранилище (S3, NFS)    |
| Пользовательские данные в памяти    | Данные в общей БД            |

В реализации:

- PostgreSQL — общая база данных (источник истины).
- Redis — распределённый кэш (общий для всех реплик).
- Аутентификация — делегирована сервису `auth` (gRPC).

Если бы сервис хранил состояние в памяти, пользователь мог бы "привязаться" к конкретной реплике, и при её падении потерял бы сессию или данные.

4. **Какие алгоритмы балансировки вы знаете?**

| Алгоритм          | Описание                                | Настройка в NGINX          |
| ----------------- | --------------------------------------- | -------------------------- |
| Round Robin       | Последовательное чередование            | По умолчанию               |
| Least Connections | Отправка к наименее загруженному        | `least_conn;`              |
| IP Hash           | Один и тот же IP всегда на одну реплику | `ip_hash;`                 |
| Weighted          | Приоритет для более мощных серверов     | `server tasks_1 weight=3;` |
| Random            | Случайный выбор                         | `random;`                  |

В данной работе: использован алгоритм Round Robin (настройка по умолчанию в NGINX).

5. **Зачем нужен endpoint `/health`?**

Endpoint `/health` выполняет несколько функций:

- Health Checks для балансировщика: NGINX (и другие балансировщики) могут опрашивать этот endpoint для определения доступных экземпляров.
- Мониторинг: инструменты observability (Prometheus, Grafana) используют его для проверки состояния сервиса и генерации алертов.
- Автоматическое восстановление: оркестраторы (Kubernetes, Docker Swarm) используют health checks для автоматического перезапуска упавших контейнеров.
- Rolling Updates: при обновлении проверяется, что новая версия отвечает на `/health` перед направлением трафика.
- Диагностика: позволяет разработчикам быстро проверить работоспособность сервиса.
