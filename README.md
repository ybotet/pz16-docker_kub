# Практическое занятие №14: Очередь задач (Job Queue) с Retries, DLQ и идемпотентностью

## Описание

Цель данной практической работы — реализовать устойчивую очередь задач (job queue) с обработкой ошибок: временные ошибки обрабатываются повторными попытками (retries), "плохие" сообщения отправляются в DLQ (Dead Letter Queue), а потребитель защищён от дублирования сообщений (идемпотентность).

## Политика повторных попыток

### Стратегия

- Максимальное количество попыток: 3.
- При ошибке увеличивается счётчик `attempt`.
- Сообщение заново публикуется в основную очередь.
- Исходное сообщение подтверждается (`ack`).
- После превышения `maxAttempts` сообщение отправляется в DLQ.

### Что считается ошибкой?

#### Permanent error (тестовый режим)
Считается ошибкой, если:
- `task_id` содержит слово `"fail"`.
- `task_id == "dlq_test"`.

#### Random error (режим нормальной работы)
- 20% вероятность случайной ошибки.

#### Имитация тяжёлой работы
- `Sleep` на 2–5 секунд.


## Dead Letter Queue (DLQ)

### Зачем нужна DLQ

- Сообщения, которые не удалось обработать после всех попыток, не теряются.
- Они не блокируют основную очередь.
- Позволяют анализировать проблемные сообщения.
- Дают возможность вручную переобработать сообщения после исправления ошибок.

## Идемпотентность

### Почему нужна идемпотентность?

Система с очередями работает в режиме **at-least-once** (как минимум один раз):

- Сообщение может быть доставлено повторно.
- Worker может упасть после обработки, но до отправки `ack`.

### Реализация

- Каждое сообщение имеет уникальный `message_id` (UUID).
- Хранилище обработанных ID в памяти (`map[string]time.Time`).
- TTL записей: 5 минут, автоматическая очистка.

```go
// idempotency/store.go
type Store struct {
    mu    sync.RWMutex
    items map[string]time.Time
    ttl   time.Duration
}

func (s *Store) IsProcessed(messageID string) bool {
    s.mu.RLock()
    defer s.mu.RUnlock()
    _, exists := s.items[messageID]
    return exists
}
```

### Проверка перед обработкой

```go
// consumer.go - processJob
if c.idempotency.IsProcessed(job.MessageID) {
    c.logger.Warn("Duplicate message detected, skipping")
    msg.Ack(false)  // Подтверждаем, но не обрабатываем
    return
}

// ......

c.idempotency.MarkProcessed(job.MessageID)
```






## Контрольные вопросы
1. Чем "job queue" отличается от "event queue"?

**Event queue** — для лёгких уведомлений (событий). Обработка быстрая, ошибки редки, повторные попытки обычно не требуются.

**Job queue** — для тяжёлых задач. Обработка может быть долгой, ошибки ожидаемы, необходимы retries, DLQ и идемпотентность.

2. Почему система очередей часто работает как at-least-once?

Из-за механизма ack: если worker обработал сообщение, но упал до отправки ack, RabbitMQ переотправит сообщение другому worker'у. Это гарантирует доставку, но может привести к дублям.

3. Как DLQ помогает эксплуатации?
- Сообщения с ошибками не теряются и не блокируют основную очередь
- Можно анализировать причины ошибок
- После исправления багов можно переобработать сообщения вручную

4. Почему ретраи нельзя делать бесконечно?

- Бесконечные retries могут замаскировать реальную проблему
- Могут привести к бесконечному циклу и перегрузке системы
- Нужен разумный лимит (обычно 3-5 попыток)

5. Что такое идемпотентность и как её реализовать минимально?

**Идемпотентность** — свойство операции, позволяющее применять её многократно без изменения результата после первого применения.

Минимальная реализация:

- Уникальный message_id в каждом сообщении
- Хранение обработанных ID (в памяти или БД)
- Проверка перед обработкой: если ID уже есть — пропустить
