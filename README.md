# Практическое занятие №11 - GraphQL API с использованием gqlgen

## Описание

Реализация **GraphQL** сервиса на Go с использованием библиотеки **gqlgen**, который предоставляет операции запросов (Query) и изменений (Mutation) для управления задачами (Tasks). Сервис подключается напрямую к базе данных **PostgreSQL**, что полностью соответствует требованиям практической работы.

## Схема GraphQL (schema.graphqls)

```graphql
type Task {
    id: ID!
    title: String!
    description: String
    done: Boolean!
    createdAt: String
    updatedAt: String
}

input CreateTaskInput {
    title: String!
    description: String
}

input UpdateTaskInput {
    title: String
    description: String
    done: Boolean
}

type Query {
    tasks: [Task!]!
    task(id: ID!): Task
}

type Mutation {
    createTask(input: CreateTaskInput!): Task!
    updateTask(id: ID!, input: UpdateTaskInput!): Task!
    deleteTask(id: ID!): Boolean!
}
```

# Описание типов

| Тип              | Описание                                                                                                       |
|------------------|----------------------------------------------------------------------------------------------------------------|
| **Task**         | Основная модель задачи. Содержит обязательные поля (`id`, `title`, `done`) и опциональные (`description`, `createdAt`, `updatedAt`). |
| **CreateTaskInput** | Input для создания задач. Определяет, какие данные клиент может отправлять при создании. |
| **UpdateTaskInput** | Input для обновления. Все поля опциональны для частичных обновлений. |
| **Query**        | Операции только для чтения (получение данных). |
| **Mutation**     | Операции записи (создание, обновление, удаление). |

# Архитектура резолверов

## Что такое резолвер

**Резолвер** — это функция, которая выполняется, когда клиент запрашивает конкретное поле в запросе GraphQL.  
Его задача — получить необходимые данные (из БД, кэша, внешнего API) и вернуть их в ожидаемом формате.

---

## Реализация резолверов

| Резолвер        | Описание                                                    | Доступ к данным                          |
|-----------------|-------------------------------------------------------------|------------------------------------------|
| **Tasks()**     | Возвращает полный список задач                              | `SELECT * FROM tasks`                    |
| **Task(id)**    | Ищет задачу по ID, возвращает `null` если не найдена        | `SELECT * FROM tasks WHERE id = $1`      |
| **CreateTask()**| Генерирует UUID, устанавливает даты, вставляет в БД         | `INSERT INTO tasks ...`                  |
| **UpdateTask()**| Изменяет переданные поля, обновляет timestamp               | `UPDATE tasks SET ... WHERE id = $1`     |
| **DeleteTask()**| Удаляет задачу, возвращает true/false                       | `DELETE FROM tasks WHERE id = $1`        |

## Внедрение зависимостей
```go
type Resolver struct {
    TaskRepo *repository.PostgresTaskRepository
    Logger   *logrus.Logger
}
```
# Репозиторий и архитектура

Репозиторий внедряется в **Resolver**, что обеспечивает:

- Разделение ответственности между GraphQL и слоем данных  
- Тестируемость (возможность мокирования репозитория)  
- Гибкость (смена БД без изменения резолверов)

## Структура таблицы `tasks`

```sql
CREATE TABLE tasks (
    id VARCHAR(36) PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    description TEXT,
    done BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

## Репозиторий PostgreSQL

Репозиторий реализует следующие операции:

```go
type PostgresTaskRepository struct {
    db     *sql.DB

// Основные методы
func (r *PostgresTaskRepository) GetAll(ctx context.Context) ([]*models.Task, error)

func (r *PostgresTaskRepository) Delete(ctx context.Context, id string) error
```

### Преимущества этой архитектуры:

- Данные сохраняются между перезапусками сервера
- GraphQL и REST используют одну базу данных (единый источник истины)
- Масштабируемость и готовность к production

## Контрольные вопросы

1. **В чём отличие Query и Mutation?**  
   Query используется для операций только чтения (получения данных). Mutation используется для операций записи, которые изменяют состояние сервера (создание, обновление, удаление).  
   Семантически — запросы представляют вопросы, которые не изменяют систему, а мутации — действия, которые действительно её меняют.  
   Технически — мутации выполняются последовательно, в то время как запросы могут выполняться параллельно.

2. **Что такое схема GraphQL и почему она считается контрактом?**  
   Схема — это полное определение всех типов, запросов, мутаций и подписок, которые GraphQL-сервер может обработать.  
   Она считается контрактом, потому что устанавливает твёрдое соглашение между клиентом и сервером: клиент точно знает, какие данные может запросить и какую структуру будет иметь ответ, а сервер обязуется соблюдать это определение.  
   Любое изменение схемы должно быть версионировано и согласовано между обеими сторонами.

3. **Что такое резолвер?**  
   Резолвер — это функция, которая выполняется, когда клиент запрашивает конкретное поле в GraphQL-запросе.  
   Его задача — получить необходимые данные из соответствующего источника (БД, внешнего API, кэша, памяти) и вернуть их в формате, ожидаемом схемой.  
   Каждое поле схемы может иметь свой собственный резолвер.

4. **Почему GraphQL решает проблему over-fetching?**  
   Over-fetching возникает, когда клиент получает больше данных, чем ему нужно, что часто происходит в REST API, где эндпоинты возвращают фиксированные структуры.  
   GraphQL позволяет клиенту указать, какие именно поля ему нужны, и сервер возвращает только эти поля.  
   Это устраняет избыточность данных, сокращает использование трафика, уменьшает время ответа и упрощает обработку данных на клиенте.

5. **Какие риски существуют при использовании GraphQL без ограничения сложности запросов?**  
   Без ограничений клиент может отправить чрезмерно вложенный или ресурсозатратный запрос (например, `друзья { друзья { друзья { ... } } }`), который потребляет большие ресурсы сервера.  
   Это может привести к **DoS-атакам**, перегрузке CPU, нехватке памяти и падению сервиса.  
   Для предотвращения таких проблем внедряются:
   - Ограничение глубины вложенности
   - Ограничение количества полей в запросе
   - Таймауты
   - Обязательная пагинация
   - Анализ стоимости запросов
